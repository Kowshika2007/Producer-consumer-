What is the Producer-Consumer Problem?

A classic Operating Systems synchronization problem where two types of processes share a fixed-size buffer. Producers add items; consumers remove them.

Synchronization Primitives

mutex_lock  Binary semaphore for mutual exclusion. Only one thread accesses the buffer at a time.

sem_empty  Counts empty slots. Initialized to buffer size. Producer blocks when 0.

sem_full   Counts filled slots. Initialized to 0. Consumer blocks when 0.

Pseudocode

# Shared state
buffer[N]
sem_empty = Semaphore(N)
sem_full  = Semaphore(0)
mutex     = Mutex()

def producer():
    item = produce_item()
    wait(sem_empty)   # block if full
    lock(mutex)       # critical section
    enqueue(item)
    unlock(mutex)
    signal(sem_full)  # notify consumer

def consumer():
    wait(sem_full)    # block if empty
    lock(mutex)       # critical section
    item = dequeue()
    unlock(mutex)
    signal(sem_empty) # notify producer
    consume(item)

Backend API (Flask)

This frontend connects to a Python Flask backend via WebSocket (socket.io). The backend runs real Python threads with actual threading.Semaphore and threading.Lock.

  • POST /api/start — start simulation
  • POST /api/stop — stop simulation
  • POST /api/reset — reset state
  • GET /api/state — current snapshot
  • WS /socket.io — real-time events

Why Semaphore Order Matters

Always wait(semaphore) before lock(mutex). Reversing causes deadlock: a producer holds the mutex and blocks on sem_empty while the consumer is blocked trying to acquire the same mutex.
